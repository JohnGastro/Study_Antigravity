# User Identity & Learning Profile

## 基本情報
- **現在のレベル**: Webエンジニアリング初学者
- **目標**: Awwwardsに掲載されるような、高品質でクリエイティブなWebサイトを作成すること。
- **参考サイト**: [Layrid](https://layrid.tomoyaokada.com/)

## 学習スタイル
- **形式**: ステップバイステップ
- **要望**:
    - 毎回 `User_identity.md` を更新して特性を記録する。
    - 毎回 `Teach_Status.md` を更新して進捗を管理する。
    - Githubを活用して「クラウドスタディネットワーク」を構築する。

## 特性メモ (随時更新)
- 初回: 意欲的で、具体的なビジュアルゴールを持っている。
- **環境**: Windows (PowerShell) から Mac (Bash/Zsh) に移行。コマンドの違い（`ni` vs `touch`）を学習中。
- **重要**: 「おせっかい」はNG。自分で手を動かして学びたいタイプ。
- **指示スタイル**: 代行するのではなく、やり方を教えてユーザーに実行させる。
- **理想の教え方**: 新しい記法や構文が出てきた時は、「答え」そのものではなく「構文のテンプレート（型）」を提示する。例: `async function 名前() { ... }` と提示し、ユーザーに当てはめさせる。「荒野に放置」せず、武器（書き方）だけ渡すスタイル。
- **学習姿勢**: 納得してから進みたいタイプ。技術選定の理由（Why）を重視する。自分のレベルとロードマップの整合性を気にする（慎重派）。
- **好みの学習形式**: クイズ・Challenge形式で、自分で考えて実装することで定着させたい。
- **振り返りの習慣**: 学習日報を書くことで、成長を可視化したいタイプ。
- **コンテキスト**: `awwwards-portfolio` は過去の遺産として無視する。`Study_Engineering` が新しい教室。
- **本質追求型**: 「なぜテキストが表示されるのか」など、根本的な仕組みを理解したい。表面的な理解では満足しない。
- **実務的視点**: 「みんなどうしてるの？」と、実際の開発現場での慣習を気にする。
- **基礎概念の説明が必要**: 「複数のclassをつける方法」など、当たり前と思われることも丁寧に説明する必要がある。「そういうこと??」と気づくタイプ。
- **写経嫌い**: 「ただ書き写すだけ」を極端に嫌う。理解していないコードを書くことにストレスを感じる。
- **思考重視**: 答えを教えられるより、ヒントをもらって自分で導き出したい。「教え方が悪い」と言われたら、答えを出しすぎているサイン。
- **理解度の不安**: コードが動いても「なぜ動くのか」が腹落ちしていないと「理解度あやし〜」と不安になる。成功体験だけでなく、納得感を重視する。
- **用語解説の即時性**: 知らない単語（`fetch`など）が出てきた場合、その文脈の中で即座に解説を入れるスタイルを好む。「後で説明」ではなく「その場で解決」がベスト。
- **比較学習**: 新しい技術（Next.js）を学ぶ際、既存の知識（Phase 2のHTML/JS）と比較することで理解が深まる。「前はこうだったけど、今はこう」という対比が効果的。
- **観察眼**: ページ遷移時のファビコンの挙動（リロードの有無）など、細かい挙動の違いに気づく鋭さがある。
- **NG**: 理由もわからずコードを書かされる「写経」は「くそ」と感じる。納得感が必須。
- **ユーティリティ思考**: Tailwindでは「共通クラスのテンプレ + 差分上書き」で考えると安心。まず構造を揃えてから個性を足すのが好み。
- **インタラクション志向**: `group` / `group-hover` など親子で連動する仕組みを図解レベルで理解してから実装したい。仕組みが腹落ちすると一気に応用できる。
- **視覚的検証派**: 余白やアニメーションは仮の背景・ボーダーを足してでも目で確認したいタイプ。見た目で確かめると理解が定着する。
- **宣言的思考**: Framer Motion の `initial` → `animate` のように「状態の変化を書くだけ」で動きが生まれるアプローチが好み。Tailwind のユーティリティ思考と似ていて理解しやすい。
- **エラー解決力**: `Module not found` や Server/Client コンポーネントのエラーを自分で原因を特定し、解決策を試行錯誤できる。エラーメッセージを読んで理解しようとする姿勢が強い。
- **違いへの敏感さ**: 新しい技術を学ぶ際、「前の技術と何が違うのか」を即座に指摘できる。Framer Motion と GSAP の違いを「面白くない、変わらない」と指摘し、技術の特徴を活かそうとする姿勢。
- **基礎から丁寧に**: 「どこに書くのか、それすらわからないレベル」と正直に伝えられる。基礎概念（React Hooks の位置など）を最初から明確に説明してもらうと理解が深まる。
